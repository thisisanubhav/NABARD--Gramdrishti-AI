# GramDrishti AI
### AI-powered foresight for rural enterprises

A NABARD-hackathon platform that predicts the future financial health of
rural micro-enterprises, detects financial stress before it happens,
explains why, and recommends what to do about it — with one portal for the
enterprise owner and one dashboard for the NABARD field officer.

*"Gram" (village) + "Drishti" (vision/foresight) — seeing financial trouble
coming before it arrives, at the village level.*

See [docs/demo_script.md](docs/demo_script.md) for the pitch and live-demo
walkthrough.

---

## 1. Problem

India's rural economy runs on millions of micro-enterprises — SHGs, FPOs,
dairy and poultry farms, food processing units, handicraft businesses, and
rural retail stores. They are economically vital but chronically underserved
by formal credit, largely because they lack the documented financial history
that lenders rely on.

At the same time, a real data trail already exists for most of these
enterprises: UPI and digital payment activity, basic income/expense records,
loan and repayment history, and — separately — public data on commodity
prices, weather, and seasonal demand. Nobody is combining these sources into
a single picture.

The result is that financial monitoring today is **manual and reactive**:
a field officer discovers an enterprise is in trouble on their next visit,
by which point the enterprise may already be facing a liquidity crisis,
a missed harvest window, or loan default. There is no system that reads the
leading indicators — a slowing income trend, rising expenses, falling
digital transaction volume, a missed repayment, an unfavorable commodity
price move — and raises a flag *before* the crisis, in language the
enterprise owner can act on.

**The gap, specifically:**
- No unified view combining financial + digital-transaction + climate/market
  data per enterprise.
- No forward-looking signal — only historical statements, if that.
- No explanation of *why* an enterprise is at risk, only that it is (when
  someone eventually notices).
- No sector-aware or region-aware context — a dairy enterprise and a
  handicrafts enterprise face structurally different risks, but are
  monitored the same way.
- No channel for turning a risk signal into a concrete next action for
  either the owner or the officer.

## 2. Our Approach

The full NABARD problem statement (see the original brief) describes a
platform broad enough to be a multi-quarter product: five sectors, climate
and market risk modeling, two full role-based applications, an alerting
system, offline support, and multilingual accessibility. Building all of it
at production depth was not the goal for a hackathon timeline — building a
**working, honest, end-to-end slice of all of it** was.

That led to a few deliberate scoping calls, made explicit so they read as
decisions rather than gaps:

| Decision | Why |
|---|---|
| **Synthetic data**, generated by a structured script rather than pulled from live APIs | Reliable for a live demo (no API keys, no rate limits, no missing-data edge cases), while still genuinely multi-source: separate financial, UPI, and climate/market tables joined by sector + state + month, exactly like real integration would work. |
| **SQLite**, not Postgres | Zero setup, file-based, portable, "works offline" by construction. Swappable later via SQLAlchemy without touching application code. |
| **Global LightGBM models**, not per-enterprise time series | Per-enterprise time-series models need years of history per enterprise to be reliable; a global model trained across all enterprise-months generalizes far better on 24 months of synthetic history, and pairs naturally with SHAP for explainability. |
| **Rule-bootstrapped risk labels, then a real classifier trained on them** | Rather than hand-waving "the AI decided," the risk definition starts as a transparent rule (cash buffer, income trend, delinquency), and a LightGBM classifier is trained on it — so it generalizes the pattern across engineered features (including climate/market ones the rule doesn't directly encode) instead of just replaying the rule. |
| **Deterministic, rule-based recommendation engine**, not an LLM call | Reliable and reproducible for a live demo — no risk of a bad generation mid-pitch — while still being personalized: template selection is driven by the same SHAP drivers that produced the alert, and further tuned by sector. |
| **Offline = IndexedDB queue + PWA installability**, not full offline-first sync | A scoped, honestly-described version of the requirement: transaction entries made offline queue locally and sync automatically on reconnect; the app shell is installable and asset-cached. This is not full offline-first data sync (read views still need connectivity), and that limit is stated rather than implied away. |
| **Static UI fully localized (EN/Hindi)**; AI-generated risk messages and recommendation text left in English | The UI chrome is translated via i18next. The risk message and recommendation text are backend-generated from deterministic templates — genuinely translatable, but only by having the backend emit translation keys instead of final strings, which is scoped as a documented follow-up rather than done partially. |

The build order followed the plan's dependency chain: data model and
synthetic data first (nothing else works without it), then the ML pipeline
(forecast + risk + explainability + recommendations, verified against known
"worsening" vs. "stable" synthetic enterprises before touching the UI), then
the two frontends, then offline/PWA/i18n polish — each slice verified against
the running system (curl smoke tests for the API, Playwright screenshots for
the UI) before moving to the next, rather than building the whole thing and
debugging at the end.

## 3. The Solution

### Enterprise Portal (`/portal`)
- Plain-language financial dashboard: current cash balance, income,
  expenses, savings.
- A color-coded risk alert banner with a plain-language explanation and an
  estimated time horizon ("may worsen within 1 month").
- A 6-month cash flow forecast chart (actual history + dashed forecast +
  confidence band).
- Personalized, sector-aware recommendations tied to the actual risk
  drivers, not generic advice.
- Simple income/expense/savings entry form — works offline, queues locally,
  and syncs automatically on reconnect.

### Field Officer Dashboard (`/officer`)
- Portfolio-level KPIs: total enterprises monitored, count by risk level.
- Average risk score by sector, so systemic sector-level stress is visible
  at a glance.
- A prioritized intervention list — every monitored enterprise, sorted
  automatically by risk level and score, filterable by sector and risk —
  so the officer's next visit is obvious without manual triage.
- Per-enterprise drill-down: same forecast, risk, and recommendation views
  as the owner sees, plus loan details (principal, outstanding, repayment
  status, missed payments).

### AI / ML capabilities
- **Cash flow forecasting** — 6-month recursive forecast with a confidence
  band, from a LightGBM regressor.
- **Financial risk classification** — Low / Medium / High, from a LightGBM
  classifier trained on rule-bootstrapped labels.
- **Early warning** — a risk score, level, and estimated time-to-stress
  horizon, computed from the same feature set as the forecast.
- **Explainable AI** — SHAP values surface the top factors behind each
  prediction in plain language (e.g. "Income trend over the last 3 months,"
  "Missed loan repayments in the last 6 months"), and the same factors drive
  the recommendations, so the explanation and the advice are always
  consistent with each other.
- **Recommendation engine** — 1-3 concrete, sector-aware actions per
  enterprise, derived from its actual top risk drivers.

### Sector intelligence
Sector is a first-class model feature and a first-class recommendation
dimension — dairy and poultry get feed-cost-specific guidance, handicrafts
gets festival-demand guidance, rural retail gets inventory/UPI-trend
guidance, across the five sectors in scope (dairy, poultry, food processing,
handicrafts, rural retail).

### Climate & market intelligence
Rainfall, temperature, commodity price index, demand index, and
flood/drought/festival flags are generated per sector+state+month, joined
into every prediction, and directly show up as SHAP-explained risk drivers
(e.g. drought conditions, commodity price trends) — not a bolted-on data
source, but part of the same feature pipeline as the financial data.

### Offline & accessibility
- IndexedDB-backed offline queue for transaction entry, with automatic sync
  on reconnect (verified: entries made offline show a "queued" indicator and
  flush automatically once back online).
- Installable PWA (manifest + service worker, cached app shell).
- Full English/Hindi localization of the UI chrome, with a persistent
  language switcher.

### 4.3 Data architecture

Core tables (`backend/app/models.py`):

- **`enterprises`** — id, name, sector, state, district, years in operation, size.
- **`financial_records`** — one row per enterprise-month: income, expenses,
  cash balance, working capital, savings. Upserted by the transaction entry
  endpoint (income/expense/savings entries fold into the current month's row).
- **`loans`** — principal, outstanding, monthly repayment, repayment status,
  missed payments in the last 6 months.
- **`upi_transactions`** — one row per enterprise-month: transaction volume,
  count, average value.
- **`external_indicators`** — a *shared* table keyed by (sector, state,
  month): rainfall, temperature, commodity price index, demand index,
  flood/drought/festival flags. Generated for 24 months of history **plus 6
  months forward**, so the forecast model has projected climate/market
  conditions to read for future months, not just historical ones.
- **`risk_alerts`**, **`users`** — alert history and role-based accounts.

All financial/UPI/external tables join on `(enterprise_id or sector+state,
month)`, which is what makes the "multi-source integration" real rather than
cosmetic: the ML feature pipeline performs actual joins across these tables,
not a single flattened mock table.

### 4.4 Synthetic data generation

`backend/app/data_gen/generate.py` produces 60 enterprises (12 per sector ×
5 sectors) across 5 states, each with 24 months of financial/UPI history
generated from sector-specific base income ranges, seasonality (e.g. dairy
winter peak, handicrafts festival peak), and noise — modulated by the
external indicators for that enterprise's sector+state+month (e.g. a
drought flag suppresses dairy/food-processing income; a festival flag boosts
handicrafts/retail demand). Ten enterprises are deliberately scripted with a
worsening trend in the final months (income decline, rising expense ratio,
falling digital adoption, loan delinquency) so the risk model and alert
system have real, verifiable positive cases rather than a hand-picked demo
value.

### 4.5 ML pipeline (`backend/app/ml/`)

```
features.py        Shared feature engineering — used by both training
                    (vectorized pandas over the full panel) and inference
                    (RecursiveState: a rolling per-enterprise window used
                    to recompute the same features at each forecast step)

forecast_model.py   LightGBM regressor, trained to predict next-month
                    cash-balance delta (not raw balance — deltas are far
                    more stationary across enterprises of different scale).
                    Applied recursively for a 6-month horizon; each step's
                    prediction feeds back in as the next step's lag feature.
                    Confidence band from the residual distribution observed
                    during training, widened at each step out.

risk_model.py       LightGBM multiclass classifier (Low/Medium/High), trained
                    on rule-bootstrapped labels (cash buffer, income trend,
                    expense ratio, delinquency) so it learns to generalize
                    the pattern across the full feature set — including
                    climate/market features the seed rule doesn't use directly.
                    SHAP TreeExplainer values against the High-risk class
                    probability give a consistent "what's pushing this
                    enterprise toward risk" explanation regardless of which
                    level is ultimately predicted. Risk score is an
                    expected-severity value (probability-weighted across
                    Low/Medium/High) rather than raw P(High), so a
                    confidently-Medium case shows a mid-range score instead
                    of collapsing toward zero.

recommend.py        Deterministic (driver, sector) → recommendation template
                    lookup. Only fires on drivers that actually increase
                    risk (a protective factor is never reported as a cause).

train.py            CLI: builds the training panel from the seeded DB,
                    trains both models, prints validation metrics, writes
                    artifacts to app/ml/artifacts/.

inference.py         Loads trained artifacts once per process, builds a
                    RecursiveState from an enterprise's live DB state, and
                    exposes forecast_for_enterprise / risk_for_enterprise /
                    recommendations_for_enterprise to the API routers.
```

Engineered features include lagged and rolling income/expenses, income
trend, expense-to-income ratio, cash buffer (months of runway), UPI
transaction trend, loan delinquency/default flags, sector and size one-hot
encoding, seasonal month encoding, and the joined external indicators.

### 4.6 API surface (`backend/app/routers/`)

| Router | Endpoints |
|---|---|
| `auth_router` | `POST /auth/register`, `POST /auth/login`, `GET /auth/me` |
| `enterprises` | `GET/POST /enterprises`, `GET /enterprises/{id}`, `GET /enterprises/{id}/financials`, `/loans`, `/upi` |
| `transactions` | `POST /enterprises/{id}/transactions` (income/expense/savings, upserts current month), `POST /enterprises/{id}/loans` |
| `intelligence` | `GET /enterprises/{id}/forecast`, `/risk`, `/recommendations` |
| `officer` | `GET /officer/enterprises` (filterable by sector/risk/state, sorted by priority), `GET /officer/summary` (portfolio KPIs + sector breakdown) |
| `external` | `GET /external/indicators?sector=&state=` |

All non-auth routes require a JWT; `enterprise_owner` accounts are scoped to
their own enterprise, `field_officer` accounts see everything.

### 4.7 Frontend architecture (`frontend/src/`)

```
api/        Typed API client (Axios + interceptors for JWT and 401 handling)
            and TanStack Query hooks (one hook per endpoint, with cache
            invalidation wired to transaction/loan mutations)

auth/       AuthContext (JWT + user in localStorage), RequireRole route
            guard, LoginPage

components/ Shared UI: CashFlowChart, RiskGauge, RiskBadge, AlertBanner,
            SectorBarChart, RecommendationCard, KpiCard, Layout,
            LanguageSwitcher

portal/     Enterprise Portal: PortalDashboard, TransactionForm

officer/    Field Officer Dashboard: OfficerDashboard (list + KPIs +
            sector chart), EnterpriseDetail (drill-down)

offline/    Dexie schema (offline transaction queue) + useOfflineQueue /
            useOnlineStatus hooks — enqueues on failure or while offline,
            auto-flushes on reconnect

i18n/       i18next config + en.json / hi.json translation resources

theme.ts    Chart color tokens (status colors reserved for risk levels,
            categorical colors assigned per sector, following the
            data-viz palette method)
```

State management is deliberately minimal: TanStack Query owns all server
state (with cache invalidation on mutation, so e.g. adding a transaction
immediately refreshes the forecast/risk/recommendation views), React context
owns only auth, and there is no separate client-side global store.

### 4.8 Verification approach

Every layer was verified against the running system rather than assumed
correct from code review alone:
- Backend: seed → train → curl smoke tests against known "worsening" vs.
  "stable" synthetic enterprises, confirming risk/forecast/recommendation
  outputs actually differ in the expected direction.
- Frontend: driven with a headless-Chromium (Playwright) script — logged in
  as each role, screenshotted every major view, toggled network offline to
  confirm the transaction queue and auto-sync behavior, and switched
  languages to confirm localization — with console errors checked on every
  pass.
- One real bug was caught this way: risk messages were citing the single
  largest-magnitude SHAP driver as "the cause" even when that driver was
  protective (negative direction), which was misleading in a genuinely
  Medium-risk, low-single-driver case. Fixed to only cite drivers that
  actually increase risk.

## 5. Running it

### Backend

```bash
cd backend
python -m venv venv
venv/Scripts/python.exe -m pip install -r requirements.txt   # venv/bin/... on macOS/Linux
venv/Scripts/python.exe -m app.data_gen.seed                 # generates & loads synthetic data
venv/Scripts/python.exe -m app.ml.train                      # trains forecast + risk models
venv/Scripts/python.exe -m uvicorn main:app --port 8000
```

### Frontend

```bash
cd frontend
npm install
npm run dev   # http://localhost:5173
```

### Demo logins (password `demo1234` for all)

- Field officer: `officer@nabard.demo`
- Enterprise owner (at-risk case): `owner1@nabard.demo`
- Enterprise owner (stable case): `owner2@nabard.demo`

Re-running the seed script wipes and regenerates the database (60 synthetic
enterprises, 24 months of history); re-run `app.ml.train` afterward to
retrain the models against the new data.

## 6. Known limitations / next steps

- AI-generated risk messages and recommendation text are English-only (see
  §2) — translating them requires the backend to emit translation keys
  instead of final strings.
- No loan entry form in the Enterprise Portal UI yet (the API endpoint
  exists; only income/expense/savings has a form).
- No enterprise self-registration UI (the API endpoint exists; only the 60
  seeded demo enterprises are reachable through the app).
- Verified at desktop viewport widths only; mobile/responsive layout has not
  been checked, despite rural users being predominantly phone-first.
- No automated test suite — verification so far has been manual
  (curl + Playwright scripts) rather than checked-in tests.
- Not yet a git repository — no version history exists yet.
